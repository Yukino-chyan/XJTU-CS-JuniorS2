## 一、实验目的与环境

### 1.1 实验目的

本实验围绕"数据结构与算法可视化"主题，设计并实现一套基于 Web 的前后端分离系统，重点训练以下能力：

1. **前后端分离架构设计**：掌握"前端交互层 + 后端服务层 + 算法执行层"三层架构的搭建与模块间交互逻辑。
2. **WebSocket 实时通信**：深入理解 WebSocket 协议，设计并实现基于 Socket.IO 的双向实时通信机制，使算法执行步骤能够实时推送至前端。
3. **算法步骤拆解**：将堆的创建（最大堆）、快速排序、二叉树遍历等算法的执行过程拆解为标准化的分步数据。
4. **前端可视化渲染**：利用 SVG 绘制树形结构和柱状图，并借助 JavaScript 动画 API（`requestAnimationFrame`）实现节点交换、柱子滑动等动画效果。
5. **工程实践**：在真实服务器上完成部署，解决版本兼容、跨域配置等工程问题。

### 2.3 WebSocket 通信流程

```
前端                                    后端
 │                                        │
 │──── connect ────────────────────────▶ │ 建立 WebSocket 连接，创建 session
 │◀─── connected {sid} ─────────────────  │
 │                                        │
 │──── start_visualization {algo,data} ▶  │ 校验数据，生成全部步骤，存入 session
 │◀─── visualization_ready {total} ─────  │ 返回总步骤数和算法类型
 │                                        │
 │──── get_step {} ────────────────────▶  │ 取 session.steps[step_index] 返回
 │◀─── step {step_data} ─────────────────  │
 │  [renderWithAnimation(step)]           │
 │  [setTimeout → get_step]              │
 │──── get_step {} ────────────────────▶  │ 重复，直至所有步骤发送完
 │◀─── visualization_complete ──────────  │
 │                                        │
 │──── pause {} ───────────────────────▶  │ session.is_paused = True
 │◀─── paused {} ────────────────────────  │
 │                                        │
 │──── resume {} ──────────────────────▶  │ session.is_paused = False
 │◀─── resumed {} ───────────────────────  │
 │                                        │
 │──── reset {} ───────────────────────▶  │ 清空 session 状态
 │◀─── reset {} ─────────────────────────  │
```

> **设计特点**：采用"前端主动拉取（get_step）"而非"后端定时推送"模式。前端渲染完动画后再请求下一步，天然规避了后端推送速度与前端渲染速度不匹配导致的帧堆积问题。`generation` 计数器防止重置后异步回调继续执行旧的步骤请求。

### 2.4 算法步骤拆解设计

分步数据采用统一的 JSON 格式，所有算法共享以下基础字段：

```json
{
  "step_id": 1,
  "algorithm": "heap",
  "description": "步骤文字描述",
  "action": "compare | swap | complete | ...",
  "color_state": "normal | comparing | swapping | completed"
}
```

各算法在此基础上附加专用字段：

| 算法 | 专用字段 |
|------|---------|
| 堆 | `array_state`、`highlight_indices`、`current_node`、`swapped_with` |
| 快速排序 | `array_state`、`pivot_index`、`left_pointer`、`right_pointer`、`swap_indices`、`partition_range` |
| 二叉树 | `tree_data`、`current_node`、`highlight_nodes`、`visited_values`、`traversal_type` |

### 2.5 前端可视化设计思路

**树形结构（堆 / 二叉树）**：按完全二叉树的层序索引计算每个节点的 SVG 坐标。设深度为 `d`，节点在本层的序号为 `p`，最大深度为 `maxDepth`，则：

```
x = (p * slots + slots/2) * (W / totalSlots)
y = topPad + d * levelH
```

每个节点渲染为带数值的圆形 `<g>` 元素，附加 `data-node-idx` 属性供动画查询。交换动画使用 `requestAnimationFrame` + ease-in-out-quad 缓动函数实现节点飞行效果。

**柱状图（快速排序）**：每根柱子高度正比于数值，渲染为带 `data-bar-idx` 的 `<g>` 元素。交换动画通过水平方向的位移插值实现柱子滑动效果。

**颜色规范**：

| 状态 | 颜色 | 含义 |
|------|------|------|
| 默认 | `#3498db`（蓝） | 未参与操作的节点 |
| 比较中 | `#f39c12`（橙） | 正在比较的节点 |
| 交换中 | `#e74c3c`（红） | 正在交换的节点 |
| 基准（pivot） | `#9b59b6`（紫） | 快速排序当前基准元素 |
| 已完成 | `#27ae60`（绿） | 操作结束 |

## 三、核心逻辑解析

### 3.1 堆创建算法步骤拆解

**算法思路**：Floyd 建堆法——从最后一个非叶子节点（索引 `⌊n/2⌋ - 1`）开始，向根节点方向依次对每个节点执行"下沉"（sift-down）操作。

**步骤拆解流程**：

```
build_max_heap():
  1. 记录初始数组状态 → action: "init"
  2. 输出最后非叶子节点索引 → action: "info"
  3. for idx in range(⌊n/2⌋-1, -1, -1):
       记录"开始调整idx子树" → action: "subtree_start"
       调用 _heapify(arr, n, idx):
         记录"开始调整节点idx" → action: "adjust_start"
         记录"比较父节点与左子" → action: "compare"
         记录"比较父节点与右子" → action: "compare"
         if largest != i:
           记录"交换节点i和largest" → action: "swap"
           执行交换
           记录"交换完成" → action: "swap_complete"
           递归 _heapify(arr, n, largest)
         else:
           记录"节点无需调整" → action: "no_change"
       记录"idx子树调整完成" → action: "subtree_complete"（高亮整棵子树）
  4. 记录"最大堆构建完成" → action: "complete"（全部变绿）
```

`subtree_complete` 步骤的引入解决了"完成状态显示时机"的问题：前端对该 `action` 的响应只是将该子树所有节点**单帧**渲染为绿色，下一步骤到来时颜色即被覆盖，不会持续残留。

### 3.2 快速排序步骤拆解

采用 Lomuto 分区方案（选最右元素为 pivot），双指针 `i`（左区右边界）和 `j`（扫描指针）：

```
_partition(arr, low, high):
  pivot = arr[high]，记录 → action: "select_pivot"
  i = low - 1
  for j in range(low, high):
    记录"扫描 arr[j]：< pivot 还是 >= pivot" → action: "compare"
    if arr[j] < pivot:
      i++
      if i != j:
        记录"与arr[i]交换，左区扩展" → action: "swap"
        执行交换
        记录"交换完成" → action: "swap_complete"
  if final_pos != high:
    记录"将pivot移至最终位置" → action: "place_pivot"
    执行交换
  记录"pivot就位" → action: "partition_complete"
  return i+1
```

前端渲染时，`swap` / `swap_complete` / `place_pivot` 步骤触发柱子水平滑动动画；`compare` 步骤高亮 `compare_index` 和 `left_pointer` 对应的柱子为橙色。

### 3.3 二叉树遍历步骤拆解

从输入数组按层序构建完全二叉树后，直接执行遍历，**跳过建树步骤**（建树过程不记录分步数据），使演示更简洁。

三种遍历的步骤记录逻辑：

```
前序：根 → 左 → 右
  访问节点时立即记录 → action: "visit", color_state: "comparing"

中序：左 → 根 → 右
  递归完左子树后记录当前节点

后序：左 → 右 → 根
  递归完左右子树后记录当前节点

遍历完成后记录 → action: "complete", visited_values: [...]
```

`visited_values` 字段随每步累积，前端在步骤展示区实时渲染遍历序列。

### 3.4 前端主循环设计

`main.js` 维护一个基于 `setTimeout` 的步进循环，而非独立的 `setInterval`：

```javascript
function scheduleNextStep() {
    stopTimer();
    if (!state.isRunning || state.isPaused) return;
    state.stepTimer = setTimeout(() => {
        algoSocket.emit('get_step', {});
    }, state.speed);
}

algoSocket.on('step', async (step) => {
    const gen = state.generation;
    state.currentStep++;
    await visualizer.renderWithAnimation(step, animDuration);
    if (gen !== state.generation) return;   // 防止旧回调继续执行
    scheduleNextStep();
});
```

`generation` 计数器在每次"重置"或"新演示"时自增，使所有正在 await 中的旧回调在恢复执行时自行丢弃，无需主动取消。

### 3.5 用户会话隔离

后端为每个 WebSocket 连接分配一个 `sid`（由 Socket.IO 自动生成），所有步骤数据、当前步骤索引、暂停状态均存储在 `user_sessions[sid]` 字典中，以 `threading.Lock` 保证线程安全：

```python
user_sessions = {}
sessions_lock = threading.Lock()

def get_session(sid: str) -> dict:
    with sessions_lock:
        if sid not in user_sessions:
            user_sessions[sid] = {
                "steps": [], "step_index": 0,
                "is_paused": False, "is_running": False, ...
            }
        return user_sessions[sid]
```

客户端断开连接时，`handle_disconnect()` 自动清除对应 session，释放内存。

## 五、问题与解决

### 问题1：快速排序 `_add_step()` 参数名错误

**现象**：提交快速排序请求时后端报错 `QuickSortRunner._add_step() got an unexpected keyword argument 'right_pointer'`，导致演示无法启动。

**原因定位**：`_add_step()` 方法的形参定义为中文名 `left指针` / `right指针`，而调用处传入的是英文关键字参数 `left_pointer` / `right_pointer`，Python 不支持中文标识符作为 `**kwargs` 的键，参数名不匹配导致 `TypeError`。

**解决方案**：将 `_add_step()` 的参数名统一修改为全英文 `left_pointer` / `right_pointer`，与调用处保持一致。

---

### 问题2：快速排序柱子只显示蓝色和紫色，无橙色/红色

**现象**：快速排序演示运行正常，但除基准元素（紫色）外，其他柱子始终保持蓝色，比较中和交换中的高亮颜色无法显示。

**原因定位**：前端 `render()` 方法中，快速排序的高亮索引通过 `step.highlight_indices` 字段读取，而后端快速排序步骤数据**从未设置此字段**（快排使用 `compare_index`、`left_pointer`、`right_pointer`、`swap_indices` 等专用字段）。字段名不匹配导致高亮列表始终为空。

**解决方案**：在前端 `render()` 中，针对 quicksort 算法从实际字段推导高亮索引：

```javascript
let hl = [];
if (action === 'swap' || action === 'swap_complete' || action === 'place_pivot') {
    hl = step.swap_indices || [];
} else if (step.compare_index != null) {
    hl = [step.compare_index];
    if (step.left_pointer != null) hl.push(step.left_pointer);
} else if (step.left_pointer != null || step.right_pointer != null) {
    hl = [step.left_pointer, step.right_pointer].filter(x => x != null);
}
```

---

### 问题3：堆的绿色子树高亮持续时间过长

**现象**：某棵子树调整完成后变绿，但绿色持续显示至下一个不相关节点开始比较，造成视觉混乱，绿色范围越来越大，难以跟踪当前操作位置。

**原因分析**：最初的设计使用持久化的 `_heapDone` 集合记录已完成的子树，每帧渲染时将集合内的节点标绿。这导致绿色节点累积不消退。

**解决方案**：改为在后端**精确地**在每次 `_heapify` 返回后立即插入一条 `subtree_complete` 步骤，`highlight_indices` 包含该子树的全部节点索引。前端对此 `action` 的响应只是**单帧**将这些节点渲染为绿色——下一步骤到来时，`_renderTree()` 会重新绘制整个 SVG，绿色自然消退。这样绿色仅"闪烁"一帧，既传递了"该子树已完成"的信息，又不影响后续步骤的颜色展示。

---

### 问题4：服务器部署 Socket.IO 版本不兼容

**现象**：将项目部署至阿里云服务器并通过 `nohup python3 app.py` 启动后，访问页面时浏览器控制台报错 `Unsupported version of Socket.IO or Engine.IO protocols`，前端无法建立 WebSocket 连接。

**原因定位**：服务器 Python 环境为 3.6，无法安装 flask-socketio 5.x（要求 Python 3.7+）。服务器虚拟环境中安装的是旧版 `flask-socketio 5.x + python-socketio 4.x + python-engineio 3.x` 的混搭组合，其中 python-engineio 3.x 实现的是 Engine.IO **协议第3版**，而前端使用的 Socket.IO **客户端 4.7.2** 需要 Engine.IO 协议第4版，版本不匹配导致握手失败。

**解决方案**：分两步修复版本链：

1. **降级后端依赖**至完全兼容 Python 3.6 的版本组合：
   ```bash
   /home/webapp/venv/bin/pip install \
       flask-socketio==4.3.2 \
       python-socketio==4.6.1 \
       python-engineio==3.14.2
   ```

2. **降级前端 Socket.IO 客户端**至与 Engine.IO 协议第3版匹配的 2.x 版本（在服务器上直接修改 `index.html` 的 CDN 链接至 Socket.IO 2.4.0）。

---

### 问题5：Windows PowerShell 中 `scp` 命令不可用

**现象**：在 PowerShell 中执行 `scp` 命令向服务器传输文件时，提示 `scp : The term 'scp' is not recognized`。

**原因**：Windows 系统 PowerShell 默认不附带 `scp`，需要通过 OpenSSH 或 Git Bash 使用。

**解决方案**：改用 Git Bash 终端执行 `scp` 命令，Git for Windows 内置了完整的 OpenSSH 工具集（包括 `scp`、`ssh`、`sftp`），可正常连接远程服务器传输文件。

## 六、实验总结与展望

### 6.1 实验收获

通过本次实验，完整实现了一套基于 Web 的算法可视化演示系统，在技术层面获得了以下收获：

1. **WebSocket 实战经验**：深入理解了 Socket.IO 的事件模型、会话管理机制，以及前端主动拉取与后端推送两种架构的权衡。"前端主动 `get_step`"的设计使渲染与数据获取天然解耦，彻底避免了帧积压问题。

2. **算法可视化设计思路**：掌握了将抽象算法拆解为分步数据的方法——关键在于**在每次状态改变的地方插入步骤记录**，而非仅记录最终结果。统一的 JSON 步骤格式使前端可以通用解析所有算法的数据。

3. **SVG 动画编程**：学会了用 `requestAnimationFrame` + 缓动函数实现平滑动画，以及利用 `data-*` 属性在 SVG 中查找特定元素的技巧。

4. **工程问题排查**：版本兼容问题的定位过程加深了对 Python 包版本链依赖关系的理解，也认识到"前后端协议版本必须配套"这一工程实践要点。

5. **服务器部署**：了解了 `nohup` 后台进程管理、阿里云安全组端口配置、`scp` 文件传输等 Linux 服务器操作技能。

### 6.2 系统不足

1. **动画回溯缺失**：目前只能前进，无法"返回上一步"，调试和观察算法细节时不够便利。

2. **多用户并发能力有限**：虽然通过 `sid` 实现了会话隔离，但服务器使用单进程 eventlet，在高并发下性能有限。

3. **快速排序分区范围标注不够直观**：当前仅通过颜色区分活跃分区，未在画布上显示分区边界线，分区范围不够一目了然。

4. **移动端兼容性**：页面布局未针对小屏幕设备优化，在手机端显示效果不理想。

### 6.3 可优化方向

1. **新增回溯功能**：在前端维护已接收步骤的历史队列，实现"上一步"按钮，允许用户反向回溯。

2. **增加更多算法**：可扩展归并排序、图的 BFS/DFS 遍历、AVL 树旋转等，进一步丰富算法库。

3. **动画速度精细控制**：在当前"每步间隔"的基础上，将动画本身的持续时间也开放为可调参数。

4. **响应式布局优化**：通过 CSS Grid / Flexbox 适配不同屏幕尺寸，提升移动端体验。

5. **生产化部署**：使用 Nginx 反向代理 + Gunicorn/uWSGI 替代直接运行 Flask 开发服务器，提升稳定性和并发能力。
